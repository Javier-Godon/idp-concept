## 1. Prerequisites

Before installing plugins, ensure the Backstage instance is running. Use the `framework/templates/backstage.k` template:

```kcl
import framework.templates.backstage as bs
import framework.templates.postgresql as pg

# PostgreSQL for Backstage (via CloudNativePG)
_pg_spec = pg.CNPGClusterSpec {
    name = "backstage-db"
    namespace = "backstage"
    instances = 2
    storageSize = "10Gi"
}

# Backstage instance
_bs_spec = bs.BackstageHelmSpec {
    name = "backstage"
    namespace = "backstage"
    host = "backstage.example.com"
    postgresHost = "backstage-db-rw.backstage.svc"
    postgresSecretName = "backstage-db-app"
    cpuRequest = "500m"
    memoryRequest = "1Gi"
}
```

**Required infrastructure**:
- Kubernetes cluster with ingress controller
- CloudNativePG operator installed
- cert-manager for TLS (optional but recommended)
- Keycloak instance (for Phase 9.2 auth)

## 2. Core Plugin Installation (Tier 1)

These plugins should be installed on Day One for a functional developer portal.

### 2.1 Kubernetes Plugin (Backstage Core)

**Purpose**: View pods, deployments, services, and other K8s objects directly in Backstage entity pages.

**Installation**:
```bash
# From your Backstage app directory
yarn --cwd packages/backend add @backstage/plugin-kubernetes-backend
yarn --cwd packages/app add @backstage/plugin-kubernetes
```

**Backend configuration** (`packages/backend/src/index.ts`):
```typescript
import { createBackend } from '@backstage/backend-defaults';
const backend = createBackend();
// ... other plugins
backend.add(import('@backstage/plugin-kubernetes-backend/alpha'));
backend.start();
```

**app-config.yaml**:
```yaml
kubernetes:
  serviceLocatorMethod:
    type: multiTenant
  clusterLocatorMethods:
    - type: config
      clusters:
        - name: production
          url: https://kubernetes.default.svc
          authProvider: serviceAccount
          serviceAccountToken: ${K8S_SA_TOKEN}
          skipTLSVerify: false
```

**Entity annotation** (auto-generated by `kcl_to_backstage`):
```yaml
metadata:
  annotations:
    backstage.io/kubernetes-id: <component-name>
```

### 2.2 TeraSky Kubernetes Ingestor

**Purpose**: Auto-create Backstage catalog entities from deployed K8s resources. Eliminates manual catalog registration — deployed workloads appear automatically.

**Installation**:
```bash
yarn --cwd packages/backend add @terasky/backstage-plugin-kubernetes-ingestor
```

**Backend configuration**:
```typescript
backend.add(import('@terasky/backstage-plugin-kubernetes-ingestor'));
```

**app-config.yaml**:
```yaml
kubernetesIngestor:
  components:
    enabled: true
    taskRunner:
      frequency:
        minutes: 5
      timeout:
        minutes: 3
    customAnnotationFilters:
      - key: "koncept.io/generated-by"
        value: "idp-concept"
    ownerAnnotation: "terasky.backstage.io/owner"
    systemAnnotation: "terasky.backstage.io/system"
    lifecycleAnnotation: "terasky.backstage.io/lifecycle"
    componentTypeAnnotation: "terasky.backstage.io/component-type"
  crossplane:
    enabled: true
    xrds:
      publishAsTemplates: true
      apiGroup: "koncept.bluesolution.es"
    claims:
      publishAsResources: true
```

**How it works with idp-concept**:
1. `kcl_to_backstage` generates `catalog-info.yaml` with correct annotations
2. `build_deployment` injects `backstage.io/kubernetes-id` annotations into K8s manifests
3. The Ingestor discovers deployed K8s workloads matching annotation filters
4. Entities appear automatically in the Backstage catalog
5. Crossplane XRDs are auto-published as Backstage Templates

### 2.5 Catalog Graph Plugin (SDA SE)

**Purpose**: Visualize entity relationships — dependency graphs showing how components, resources, and systems relate to each other.

**Installation**:
```bash
yarn --cwd packages/app add @backstage/plugin-catalog-graph
```

**Entity page integration**:
```typescript
import { EntityCatalogGraphCard } from '@backstage/plugin-catalog-graph';

// Add to the entity layout
<EntityLayout.Route path="/graph" title="Dependencies">
  <EntityCatalogGraphCard variant="gridItem" height={400} />
</EntityLayout.Route>
```

**How it maps to idp-concept**:
- Component → `dependsOn` → Resource (e.g., erp-api → erp-postgres)
- System → `hasPart` → Component/Resource (e.g., ERP Back → erp-api, erp-postgres)
- Domain → `hasPart` → System (e.g., ERP Back → ERP Back system)

## 3. Auth & RBAC via Keycloak

### 3.1 Keycloak Instance

Use the existing Keycloak template from `framework/templates/keycloak.k`:

```kcl
import framework.templates.keycloak as kc

_kc_spec = kc.KeycloakSpec {
    name = "backstage-keycloak"
    namespace = "auth"
    instances = 2
    hostname = "auth.example.com"
    dbHost = "keycloak-db-rw.auth.svc"
    dbSecretName = "keycloak-db-app"
}
```

### 3.2 Keycloak Realm Configuration

Create a `backstage` realm with the following clients and roles:

**Keycloak Client**:
```json
{
  "clientId": "backstage",
  "protocol": "openid-connect",
  "publicClient": false,
  "redirectUris": ["https://backstage.example.com/api/auth/keycloak/handler/frame"],
  "webOrigins": ["https://backstage.example.com"]
}
```

**Roles**:
| Keycloak Role | Backstage Permission | Description |
|---|---|---|
| `developer` | Read-only catalog + template usage | Can view catalog, use scaffolder |
| `platform-engineer` | Full catalog + admin access | Can edit entities, manage templates |
| `manager` | Read-only catalog + metrics | Can view catalog and dashboards |

### 3.3 Backstage Keycloak Plugin Installation

**Installation** (Red Hat Keycloak plugin):
```bash
yarn --cwd packages/backend add @backstage-community/plugin-auth-backend-module-keycloak-provider
yarn --cwd packages/app add @backstage-community/plugin-auth-provider-keycloak
```

**Backend configuration**:
```typescript
backend.add(import('@backstage-community/plugin-auth-backend-module-keycloak-provider'));
```

**app-config.yaml**:
```yaml
auth:
  environment: production
  providers:
    keycloak:
      production:
        metadataUrl: https://auth.example.com/realms/backstage/.well-known/openid-configuration
        clientId: backstage
        clientSecret: ${KEYCLOAK_CLIENT_SECRET}
        prompt: auto
```

### 3.4 Permission Framework

**app-config.yaml** (RBAC):
```yaml
permission:
  enabled: true
  rbac:
    policies:
      - effect: allow
        principal: group:developer
        permission: catalog.entity.read
      - effect: allow
        principal: group:developer
        permission: scaffolder.template.execute
      - effect: allow
        principal: group:platform-engineer
        permission: catalog.*
      - effect: allow
        principal: group:platform-engineer
        permission: scaffolder.*
      - effect: allow
        principal: group:manager
        permission: catalog.entity.read
```

## 4. TechDocs Configuration

### 4.1 Enable TechDocs

TechDocs generates documentation sites from Markdown files in the repository.

**app-config.yaml**:
```yaml
techdocs:
  builder: local
  generator:
    runIn: local
  publisher:
    type: local
    local:
      publishDirectory: /tmp/techdocs
```

For production, use external storage:
```yaml
techdocs:
  builder: local
  generator:
    runIn: local
  publisher:
    type: awsS3  # or googleGcs, azureBlobStorage
    awsS3:
      bucketName: backstage-techdocs
      region: eu-west-1
```

### 4.2 TechDocs Annotations

The `kcl_to_backstage` procedure automatically adds `backstage.io/techdocs-ref` to Domain and System entities when configured via render.k (`techdocs_ref` parameter).

**Generated annotation**:
```yaml
metadata:
  annotations:
    backstage.io/techdocs-ref: dir:docs/
```

### 4.3 mkdocs.yml Configuration

Create `mkdocs.yml` at the repository root:

```yaml
site_name: idp-concept Platform Documentation
nav:
  - Home: README.md
  - Getting Started:
      - Developer Quickstart: docs/DEVELOPER_QUICKSTART.md
      - Developer Guide: docs/DEVELOPER_GUIDE.md
  - Architecture:
      - Project Architecture: docs/PROJECT_ARCHITECTURE.md
      - Framework Schemas: docs/FRAMEWORK_SCHEMAS.md
  - Operations:
      - User Workflow Guides: docs/USER_WORKFLOW_GUIDES.md
      - Migration Guide: docs/MIGRATION_GUIDE.md
      - Tooling Setup: docs/TOOLING_SETUP.md
  - Security: docs/SECURITY.md
  - Testing: docs/TESTING_STRATEGY.md
  - Backstage:
      - Adoption Analysis: docs/BACKSTAGE_ADOPTION_ANALYSIS.md
      - Plugin Guide: docs/BACKSTAGE_PLUGIN_GUIDE.md
plugins:
  - techdocs-core
```

**Purpose**: Embed Grafana monitoring dashboards per service in entity pages.

**Installation**:
```bash
yarn --cwd packages/app add @backstage-community/plugin-grafana
```

**app-config.yaml**:
```yaml
grafana:
  domain: https://grafana.example.com
```

```yaml
metadata:
  annotations:
    grafana/dashboard-selector: "tags @> 'backstage' && tags @> '<component-name>'"
```

### 5.2 Prometheus Plugin

**Purpose**: Display Prometheus metrics and alerts per service.

**Installation**:
```bash
yarn --cwd packages/app add @roadiehq/backstage-plugin-prometheus
```

**Entity annotation**:
```yaml
metadata:
  annotations:
    prometheus.io/rule: "sum(rate(http_requests_total{service=\"<component-name>\"}[5m]))"
```

### 5.3 Kafka Plugin

**Purpose**: Monitor Strimzi Kafka clusters, topics, and consumer groups.

**Installation**:
```bash
yarn --cwd packages/app add @backstage-community/plugin-kafka
```

**app-config.yaml**:

**Purpose**: View CI/CD pipeline status directly in entity pages.

## 6. Plugin Configuration Reference

### Complete app-config.yaml Template

```yaml
app:
  title: idp-concept Portal
  baseUrl: https://backstage.example.com

organization:
  name: idp-concept

backend:
  baseUrl: https://backstage.example.com
  listen:
    port: 7007
  database:
    client: pg
    connection:
      host: backstage-db-rw.backstage.svc
      port: 5432
      user: backstage
      password: ${POSTGRES_PASSWORD}

auth:
  environment: production
  providers:
    keycloak:
      production:
        metadataUrl: https://auth.example.com/realms/backstage/.well-known/openid-configuration
        clientId: backstage
        clientSecret: ${KEYCLOAK_CLIENT_SECRET}

catalog:
  locations:
    - type: url
      target: https://github.com/org/repo/blob/main/catalog-info.yaml
      rules:
        - allow: [Domain, System, Component, Resource]

kubernetes:
  serviceLocatorMethod:
    type: multiTenant
  clusterLocatorMethods:
    - type: config
      clusters:
        - name: production
          url: https://kubernetes.default.svc
          authProvider: serviceAccount
          serviceAccountToken: ${K8S_SA_TOKEN}

kubernetesIngestor:
  components:
    enabled: true
    taskRunner:
      frequency:
        minutes: 5
      timeout:
        minutes: 3
    customAnnotationFilters:
      - key: "koncept.io/generated-by"
        value: "idp-concept"

argocd:
  baseUrl: https://argocd.example.com
  username: ${ARGOCD_USERNAME}
  password: ${ARGOCD_PASSWORD}

techdocs:
  builder: local
  generator:
    runIn: local
  publisher:
    type: local

permission:
  enabled: true
```

### Entity Annotations Cheat Sheet

| Annotation | Source | Purpose |
|---|---|---|
| `backstage.io/kubernetes-id` | `kcl_to_backstage` + `build_deployment` | K8s plugin entity matching |
| `backstage.io/techdocs-ref` | `kcl_to_backstage` (techdocs_ref param) | TechDocs documentation link |
| `backstage.io/source-location` | `kcl_to_backstage` (repo_url param) | Source code link |
| `koncept.io/generated-by` | `kcl_to_backstage` | Identifies idp-concept entities |
| `koncept.io/module-type` | `kcl_to_backstage` | Module type (APPLICATION, INFRASTRUCTURE, CRD, SECRET) |
| `argocd/app-name` | Manual or `kcl_to_argocd` | ArgoCD plugin entity matching |
| `grafana/dashboard-selector` | Manual | Grafana dashboard embedding |
| `prometheus.io/rule` | Manual | Prometheus metrics display |
| `github.com/project-slug` | Manual | GitHub Actions plugin |

## 7. Troubleshooting

### Plugin Issues

| Problem | Solution |
|---|---|
| Kubernetes plugin shows no pods | Verify `backstage.io/kubernetes-id` annotation matches the K8s label `app.kubernetes.io/name` |
| Ingestor not creating entities | Check `customAnnotationFilters` configuration. Ensure deployed resources have `koncept.io/generated-by: idp-concept` |
| ArgoCD plugin unauthorized | Verify `ARGOCD_USERNAME`/`ARGOCD_PASSWORD` or `ARGOCD_AUTH_TOKEN` env vars |
| TechDocs not rendering | Ensure `mkdocs.yml` exists at repo root with `techdocs-core` plugin |
| Keycloak login fails | Verify redirect URI matches `https://backstage.example.com/api/auth/keycloak/handler/frame` |
| Catalog Graph empty | Ensure entities have `spec.system`, `spec.dependsOn`, or `spec.owner` relations |

### Useful Commands

```bash
# Generate catalog-info.yaml
cd projects/my-project/pre_releases/manifests/dev/factory
koncept render backstage

# Validate catalog-info.yaml
npx @backstage/cli catalog-info validate catalog-info.yaml

# Test K8s plugin connectivity
kubectl auth can-i list pods --as=system:serviceaccount:backstage:backstage
```
