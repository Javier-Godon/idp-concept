# Distribution & Sharing Model

> How **idp-concept** is packaged, installed, and how team members share and consume each
> other's work. This is a decision record — it answers "what is the product, and how do we
> collaborate with it?"

## TL;DR

There are **two distinct things**, and they should not be confused:

| Concern | What it is | How it is shared |
|---|---|---|
| **The tooling** | The Go `koncept` CLI (plus the pinned KCL toolchain it drives) | Installed as a package: cross-platform binaries + checksums, or a pinned container image |
| **The work** | KCL source (projects/modules/stacks) and the rendered manifests | A shared **Git repository**, reconciled to clusters by a GitOps controller (ArgoCD) |

You install the **tooling once**. You share your **work through Git**. You do **not** pass
rendered files around by hand, and you do not "install" another team's work — you read and
build on it through the shared repository and the GitOps control loop.

## The installable package = the CLI

The single, supported interface is the Go binary in `cmd/koncept`. It is the "package that
anybody on the team can install and enjoy the whole functionality of the IDP."

- **Distribution**: `make build-all` + `make checksums` produce cross-platform binaries and a
  `SHA256SUMS` file; `.github/workflows/release.yml` publishes them as GitHub Release assets on
  `v*` tags. `make docker` builds a pinned container image (bundling the `kcl` toolchain) that
  is pushed to GHCR for CI and reproducible local use.
- **Install paths**: a single binary on `PATH` for individuals, or the pinned image for CI and
  Docker-based environments. See [TOOLING_SETUP.md](../operations/TOOLING_SETUP.md).
- **What it provides**: scaffolding (`init project|module|env|release`), rendering
  (`render <format>`), validation and governance (`doctor`, `policy check`, `golden check`,
  `changelog`), and diagnostics (`deps`, `metrics`).

The `framework/` engine is consumed as a **versioned KCL module**, not copied per project.
See [FRAMEWORK_VERSIONING.md](../platform-engineering/FRAMEWORK_VERSIONING.md).

## The shared work = a Git repository + GitOps

Collaboration happens through Git, not through file hand-off:

1. **Author** — a team member installs the CLI, writes KCL (a project, a module, a stack), and
   renders it with `koncept render <format>`.
2. **Commit** — they commit both the KCL source and the committed rendered output (the golden
   files) to the shared Git repository. CI validates: `koncept doctor`, `policy check`,
   `golden check`, and the render smoke matrix.
3. **Review** — changes are reviewed as code (see [OPERATING_MODEL.md](../operations/OPERATING_MODEL.md)
   for roles and approval paths).
4. **Reconcile** — a GitOps controller (ArgoCD) watches the repository and applies the rendered
   manifests to the target clusters. The Git repository is the single source of *intent*; the
   cluster state converges to it.
5. **Consume** — other members access your work by reading the same repository: they import
   your KCL modules, reuse your stacks, or simply depend on the services you deployed. The
   Backstage catalog (generated by `koncept render backstage`) makes this discoverable.

```
 install once                      share through Git + GitOps
┌──────────────┐   author/render   ┌───────────────┐   reconcile   ┌──────────────┐
│ koncept CLI  │ ────────────────▶ │  Git repo      │ ───────────▶ │  Clusters     │
│ (the package)│                   │  KCL + rendered│   (ArgoCD)    │ (LOCAL→PROD)  │
└──────────────┘                   │  manifests     │              └──────────────┘
        ▲                          └───────────────┘                      │
        └──────────── everyone reads/builds on the same repo ◀────────────┘
```

## Why this split

- **Reproducibility** — everyone runs the same pinned CLI + KCL toolchain, so renders are
  deterministic (golden files guard drift).
- **Auditability** — every change to intent is a reviewed Git commit; the cluster never drifts
  silently from what is in Git.
- **No lock-step coupling** — teams do not need to be online together or exchange artifacts;
  Git is the asynchronous integration point.
- **Self-service without chaos** — the CLI gives every developer the full capability, while the
  shared repo + policy gates keep the platform consistent.

## What this is NOT

- It is **not** a server you log into to "submit jobs." The CLI is client-side; the only
  server-side component is whatever GitOps/Kubernetes you already run.
- It is **not** a model where rendered YAML is emailed, Slacked, or copied between machines.
  If it is not in Git, it does not exist for the platform.
